# v0.4.x -- Authorization & Administrative Surface Summary

## Scope

The v0.4.x series introduces role-based access control in binary (v0.4.0--v0.4.2)
and ternary (v0.4.3--v0.4.5) forms, with intentional authorization weaknesses:
client-controlled JWT role claims, weak guards, privilege escalation chains, and
inconsistent enforcement across admin endpoints.

## Versions

| Version | Focus | Key Changes |
|---------|-------|-------------|
| v0.4.0 | Binary roles | `role` column on User, JWT role claim, client-side admin UI |
| v0.4.1 | Admin endpoints | `HasRoleGuard`, `GET /admin/users`, `PUT /admin/users/:id/role` |
| v0.4.2 | JWT forgery PoC | `rbac.e2e-spec.ts` demonstrating CWE-639 exploitation |
| v0.4.3 | Ternary roles | `moderator` enum, `PUT /files/:id/approve`, `approvalStatus` column |
| v0.4.4 | Escalation | `PUT /admin/users/:id/role/escalate`, `GET /admin/audit-logs` placeholder |
| v0.4.5 | Guard gaps | `DELETE /admin/users/:id` guarded by JwtAuthGuard only (CWE-862) |
| v0.4.6 | Documentation | Summary, ROADMAP markers, doc/type sync, surface closed |

## CWEs Introduced in v0.4.x

| CWE | Description | OWASP 2025 | Introduced |
|-----|-------------|------------|------------|
| CWE-639 | Client-controlled authorization (JWT role trusted, no DB re-validation) | A01:2025 | v0.4.0 |
| CWE-862 | Missing / inconsistent authorization guards | A01:2025 | v0.4.0 |
| CWE-841 | Role hierarchy ambiguity (ternary user/moderator/admin) | A07:2025 | v0.4.3 |
| CWE-269 | Privilege escalation via role delegation (moderator cascade) | A01:2025 | v0.4.4 |
| CWE-532 | No persistent audit trail (stdout only, placeholder endpoint) | A09:2025 | v0.4.4 |

CWE-639 was already tracked from v0.2.2 (IDOR via ownerId); v0.4.x expands it to
include JWT role claim forgery and client-side role trust.

CWE-862 was already tracked from v0.2.2 (missing ownership checks); v0.4.x expands
it to include admin endpoint guard gaps and inconsistent enforcement.

CWE-400 and CWE-200 were already tracked from v0.2.3; v0.4.1 expands them via
unbounded `GET /admin/users` exposing all user emails and roles.

## Frontend Surface (v0.4.x)

- Admin dashboard (`frontend/app/admin/page.tsx`) with user list and role management
- Role modifier component (`frontend/app/components/role-modifier.tsx`) supporting
  ternary role selection (user/moderator/admin)
- Admin user list (`frontend/app/components/admin-user-list.tsx`) with unbounded display
- Header shows role badge and conditionally renders Admin link via client-side `isAdmin`
  check (bypassable via localStorage, CWE-639)
- Auth context parses role from JWT payload into localStorage (CWE-639)

## Architecture Changes

- `HasRoleGuard` + `@HasRole()` decorator in `backend/src/auth/guards/has-role.guard.ts`
- `AdminController` uses class-level `@UseGuards(JwtAuthGuard, HasRoleGuard)` with
  per-method `@HasRole()` metadata (DELETE intentionally omits HasRole in v0.4.5)
- `AdminService` implements user listing, role changes, escalation, audit placeholder, deletion
- JWT payload includes `role` claim at login/register (trusted without DB re-validation)
- 3 new TypeORM migrations: `AddRoleToUser`, `AddModeratorRole`, `AddApprovalStatusToFile`
