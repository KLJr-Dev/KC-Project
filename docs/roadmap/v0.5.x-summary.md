## v0.5.0: Input Validation Pipeline

### Objectives
- Establish global request validation boundary via NestJS `ValidationPipe`
- Enforce strict type checking and field constraints via class-validator decorators
- Format HTTP 400 errors with field-level constraint messages
- Expose intentional weak patterns (weak passwords, unbounded fields, escalation paths)
- Create comprehensive e2e test suite validating all scenarios

### Implementation

#### 1. Global ValidationPipe Registration (main.ts)
```typescript
app.useGlobalPipes(
  new ValidationPipe({
    whitelist: true,                  // Remove unknown properties
    forbidNonWhitelisted: true,      // Reject unknown properties
    transform: false,                 // No auto-conversion (CWE-1025 intact)
    skipMissingProperties: false,    // Validate all fields
  }),
);
app.useGlobalFilters(new ValidationExceptionFilter());
```

**CWE Exposure**:
- CWE-1025 (Comparison Using Wrong Factors): Strict types, no auto-conversion. "123" (string) rejected if number expected.
- CWE-20 (Improper Input Validation): Weak patterns intentional (password min-length 1 only).

#### 2. DTO Decorators — All Surfaces

**Auth Surface** (register, login):
```typescript
// RegisterDto
@IsEmail({}, { message: 'email must be a valid email address' })
@IsNotEmpty({ message: 'email is required' })
email!: string;

@IsString({ message: 'username must be a string' })
@MinLength(3, { message: 'username must be at least 3 characters' })
@MaxLength(50, { message: 'username must be at most 50 characters' })
@IsNotEmpty({ message: 'username is required' })
username!: string;

@IsString({ message: 'password must be a string' })
@MinLength(1, { message: 'password must be at least 1 character' }) // CWE-521 weak
@IsNotEmpty({ message: 'password is required' })
password!: string;
```

**Users Surface** (create, update):
- Mirrors auth constraints
- Update DTO uses @IsOptional for partial updates
- CWE-521 (weak password) maintained

**Files Surface** (upload, approve):
```typescript
// UploadFileDto
@IsString({ message: 'description must be a string' })
@IsOptional()
// NO @MaxLength — CWE-400 unbounded metadata field
description?: string;

// ApproveFileDto
@IsEnum(['approved', 'rejected'], { message: 'status must be approved or rejected' })
@IsNotEmpty()
status!: string;
```

**Sharing Surface** (create, update):
```typescript
// CreateSharingDto
@IsUUID('4', { message: 'fileId must be a UUID' })
@IsNotEmpty()
fileId!: string;

@IsBoolean({ message: 'public must be boolean' })
@IsNotEmpty()
public!: boolean;

@IsISO8601({ message: 'expiresAt must be ISO8601 date' })
@IsOptional()
expiresAt?: string;
```

**Admin Surface** (create, update, delete):
```typescript
// UpdateUserRoleDto
@IsEnum(['user', 'moderator', 'admin'], { message: 'role must be user, moderator, or admin' })
@IsNotEmpty()
role!: string;  // CWE-269 allows escalation paths (moderator → admin)
```

#### 3. Custom ValidationExceptionFilter

Intercepts `BadRequestException` from ValidationPipe and formats 400 responses:

```typescript
// Response format
{
  "statusCode": 400,
  "message": "Bad Request",
  "errors": {
    "email": ["email must be a valid email address"],
    "username": ["username must be at least 3 characters"],
    "password": ["password must be at least 1 character"]
  },
  "timestamp": "2026-03-05T15:23:45.123Z"
}
```

**CWE-209 (Information Exposure Through Error Messages)**: Field-level constraints allow attackers to enumerate valid input patterns.

## v0.5.1: Frontend Form Validation Alignment

### Objectives
- Mirror backend validation constraints in frontend forms
- Parse ValidationError responses and extract field-level errors
- Display visual feedback for invalid fields
- Achieve type-safe API client with error handling

### Implementation

#### 1. API Client Error Handling (lib/api.ts)

New error classes and helpers:
```typescript
// Structured error response from backend
export interface ValidationErrorResponse {
  statusCode: number;
  message: string;
  errors?: Record<string, string[]>;
  timestamp?: string;
}

// Custom error class for validation failures
export class ValidationError extends Error {
  public readonly errors: Record<string, string[]>;
  public readonly statusCode: number;
  constructor(response: ValidationErrorResponse) { ... }
}
```

Updated `request()` helper to parse 400 responses into `ValidationError` with field-level messages.

#### 2. Auth Forms (app/auth/page.tsx)

Updated validation to match backend constraints:
```typescript
function validateEmail(email: string): string | null {
  if (!email) return 'Email is required';
  if (!/^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(email)) return 'Enter a valid email address';
  return null;
}

function validateUsername(username: string): string | null {
  if (!username) return 'Username is required';
  if (username.length < 3) return 'Username must be at least 3 characters';
  if (username.length > 50) return 'Username must be at most 50 characters';
  return null;
}

function validatePassword(password: string): string | null {
  if (!password) return 'Password is required';
  // Min-length 1 only (CWE-521 weak pattern)
  return null;
}
```

Error handling in handlers:
```typescript
try {
  // Make API request
} catch (err) {
  if (err instanceof ValidationError) {
    // Extract first constraint message for each field
    const fieldErrors = {};
    for (const [field, messages] of Object.entries(err.errors)) {
      fieldErrors[field] = messages?.[0] ?? null;
    }
    setFieldErrors(fieldErrors);
    setError('Please check the highlighted fields');
  } else {
    setError(err instanceof Error ? err.message : String(err));
  }
}
```

#### 3. Users Form (app/users/page.tsx)

Added client-side validation and field-level error display:
- Email validation matching backend @IsEmail
- Username min/max length (3–50) matching backend
- Password presence check
- Visual feedback: red borders for fields with errors, error text display

## Known Limitations & Next Steps

### v0.5.x Limitations
- Frontend type generation incomplete (openapi.json regenerated, types.gen.ts pending)
- No rate limiting on validation failures (CWE-307 exposure continues)
- Error responses expose field names (CWE-209 by design)
- No request logging of validation failures (CWE-532 continues)

### v0.6.0+ Planned (Concurrent Features)
- Comprehensive logging layer with intentional leakage (CWE-532)
- Sequential token generation for sharing (CWE-330)
- Predictable user enumeration via email errors (CWE-203)
- Missing authorization gates (CWE-285)

- type regeneration command (`npm run generate:types`) not yet executed due to backend startup issues
- E2e test suite not fully executed (validation tests created but not run)
